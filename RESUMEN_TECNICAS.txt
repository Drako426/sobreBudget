🧠 2. Chain-of-Thought (20% del proceso)
═══════════════════════════════════════════════════════════════════════════════

PROPÓSITO: Descomponer problemas complejos en pasos intermedios

Utilizacion:
├─ Diseño de la lógica del semáforo (3 estados)
├─ Sistema de almacenamiento versionado
├─ Validaciones de formularios
├─ Filtrado multi-dimensional de transacciones
└─ Algoritmos de cálculo de estadísticas

CASOS EXPLICADOS CON CoT:

1. SEMÁFORO DE GASTOS
   ├─ Paso 1: Definir el problema (% del presupuesto)
   ├─ Paso 2: Establecer umbrales (70%, 100%)
   ├─ Paso 3: Fórmula (percentage = spent / budget * 100)
   ├─ Paso 4: Implementar (if/else con 3 ramas)
   └─ Paso 5: Validar (casos de prueba: 62.5%→verde, 112.5%→rojo)

   Resultado:
   if (percentage ≤ 70%)  → 🟢 verde
   if (percentage ≤ 100%) → 🟡 amarillo
   if (percentage > 100%) → 🔴 rojo

2. ALMACENAMIENTO VERSIONADO
   ├─ Paso 1: Identificar riesgo (cambios en estructura)
   ├─ Paso 2: Diseñar claves ("sobrebudget:v1:categories")
   ├─ Paso 3: Implementar constantes de versión
   ├─ Paso 4: Crear funciones genéricas get/set
   └─ Paso 5: Beneficios (compatibilidad futura, migraciones)

3. VALIDACIÓN DE TRANSACCIONES
   ├─ Paso 1: Requisitos (monto, fecha, categoría, tipo)
   ├─ Paso 2: Reglas (amount > 0, date requerido, etc)
   ├─ Paso 3: Implementar validaciones
   ├─ Paso 4: Mensajes de error claros
   └─ Paso 5: Testing mental (casos límite)

4. FILTRADO MULTI-DIMENSIONAL
   ├─ Paso 1: Dimensiones (mes, categoría, búsqueda)
   ├─ Paso 2: Lógica (AND entre filtros)
   ├─ Paso 3: Algoritmo (filter con 3 predicados)
   ├─ Paso 4: Casos de prueba (todas las combinaciones)
   └─ Paso 5: Optimización (O(n) performance)

TOTAL DE DESCOMPOSICIONES: 4 análisis CoT
PASOS INTERMEDIOS TOTALES: ~20 pasos de razonamiento


🌳 3. Tree-of-Thoughts (10% del proceso)
═══════════════════════════════════════════════════════════════════════════════

PROPÓSITO: Explorar múltiples rutas de decisión, elegir la mejor

Utilizacion: 5 decisiones estratégicas principales

1. ALMACENAMIENTO: localStorage vs Supabase vs IndexedDB
   ├─ Rama 1: localStorage
   │  ├─ Pros:  ✅ Sin backend, sin .env, privacidad, offline
   │  ├─ Cons:  ❌ No multiplataforma
   │  └─ Score: 9/10 ⭐ SELECCIONADA
   │
   ├─ Rama 2: Supabase
   │  ├─ Pros:  ✅ Profesional, multiplataforma
   │  ├─ Cons:  ❌ Requiere .env, requiere backend
   │  └─ Score: 3/10 ❌ (incumple requisitos)
   │
   └─ Rama 3: IndexedDB
      ├─ Pros:  ✅ Más capacidad
      ├─ Cons:  ❌ Complejo, innecesario
      └─ Score: 4/10 ⚠️ (overkill)

2. GRÁFICAS: Recharts vs Chart.js vs D3.js vs Visx
   ├─ Rama 1: Recharts
   │  ├─ Pros:  ✅ React nativo, TypeScript, pequeño bundle
   │  ├─ Cons:  ❌ Menos customizable que D3
   │  └─ Score: 9/10 ⭐ SELECCIONADA
   │
   ├─ Rama 2: Chart.js
   │  ├─ Pros:  ✅ Popular, customizable
   │  ├─ Cons:  ❌ No-idiomático en React
   │  └─ Score: 6/10 ⚠️ (funciona pero no ideal)
   │
   ├─ Rama 3: D3.js
   │  ├─ Pros:  ✅ Ultra customizable
   │  ├─ Cons:  ❌ Curva de aprendizaje, bundle grande
   │  └─ Score: 2/10 ❌ (overkill)
   │
   └─ Rama 4: Visx
      ├─ Pros:  ✅ D3 idiomático en React
      ├─ Cons:  ❌ Más complejo que Recharts
      └─ Score: 7/10 ⚠️ (buena pero innecesaria)

3. TIPOS: Un archivo vs Múltiples vs Inline
   ├─ Rama 1: Todo en src/types/index.ts
   │  ├─ Pros:  ✅ Simple, fácil de encontrar
   │  ├─ Cons:  ❌ Puede crecer mucho
   │  └─ Score: 9/10 ⭐ SELECCIONADA (proyecto pequeño)
   │
   ├─ Rama 2: Múltiples archivos
   │  ├─ Pros:  ✅ Organizado, escalable
   │  ├─ Cons:  ❌ Más importes, overkill
   │  └─ Score: 5/10 ⚠️ (mejor para grandes proyectos)
   │
   └─ Rama 3: Inline
      ├─ Pros:  ✅ Colocación lógica
      ├─ Cons:  ❌ Duplicación, difícil reutilizar
      └─ Score: 2/10 ❌ (no recomendado)

4. EXPORT: JSON download vs Cloud sync vs URL encoding
   ├─ Rama 1: JSON download
   │  ├─ Pros:  ✅ Universal, portable, simple
   │  ├─ Cons:  ❌ Manual
   │  └─ Score: 9/10 ⭐ SELECCIONADA
   │
   ├─ Rama 2: Cloud sync
   │  ├─ Pros:  ✅ Automático
   │  ├─ Cons:  ❌ Requiere backend, complejo
   │  └─ Score: 2/10 ❌ (fuera de scope)
   │
   └─ Rama 3: URL encoding
      ├─ Pros:  ✅ Shareable
      ├─ Cons:  ❌ URLs largas, límite caracteres
      └─ Score: 3/10 ❌ (no práctico)

5. COMPONENTES: Anidados vs Globales+Específicos vs Monolithic
   ├─ Rama 1: Globales + Específicos
   │  ├─ Pros:  ✅ DRY, separación clara, escalable
   │  ├─ Cons:  ⚠️ Más carpetas
   │  └─ Score: 9/10 ⭐ SELECCIONADA
   │
   ├─ Rama 2: Anidados
   │  ├─ Pros:  ✅ Colocation lógica
   │  ├─ Cons:  ❌ Duplicación de código
   │  └─ Score: 5/10 ⚠️ (si hay mucha duplication)
   │
   └─ Rama 3: Monolithic
      ├─ Pros:  ✅ Simple al inicio
      ├─ Cons:  ❌ Caótico rápido
      └─ Score: 3/10 ❌ (no escalable)

TOTAL DE DECISIONES ESTRATÉGICAS: 5
RAMAS EXPLORADAS: 16 alternativas evaluadas
DECISIONES CORRECTAS: 5/5 (100%)
