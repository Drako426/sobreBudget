1️⃣ REACT: Thought → Action → Observation (70%)
═══════════════════════════════════════════════════════════════════════════════

DEFINICIÓN
ReAct alterna continuamente entre:
  • Thought: Análisis del siguiente paso
  • Action: Ejecución concreta (crear archivo, comando)
  • Observation: Verificación del resultado

APLICACIÓN EN SOBREBUDGET

Fase 1: Limpieza
├─ Thought: "Necesito un proyecto limpio desde cero"
├─ Action: rm -rf src/components src/pages src/lib src/types...
└─ Observation: Proyecto limpio, dependencias instaladas ✓

Fase 2: Tipos y Almacenamiento
├─ Thought: "¿Cómo defino los modelos de datos?"
├─ Action: Crear src/types/index.ts con Category, Transaction
├─ Observation: Tipos definidos ✓
├─ Thought: "¿Cómo persisto los datos?"
├─ Action: Crear src/lib/storage.ts con API localStorage
└─ Observation: Almacenamiento funcionando ✓

Fase 3: Lógica y Formatos
├─ Thought: "¿Cómo formato COP y fechas en es-CO?"
├─ Action: Crear src/lib/formatters.ts
├─ Observation: Formato correcto ✓
├─ Thought: "¿Cómo calculo el semáforo y estadísticas?"
├─ Action: Crear src/lib/helpers.ts
└─ Observation: Lógica de negocio completa ✓

Fase 4: Componentes
├─ Thought: "¿Qué componentes reutilizables necesito?"
├─ Action: Crear 6 componentes (Navbar, Modal, Cards, etc)
└─ Observation: Componentes reutilizables listos ✓

Fase 5: Páginas
├─ Thought: "¿En qué orden creo las páginas?"

├─ Thought: "¿Necesito router?"

└─ Observation: Router funcionando ✓

├─ Thought: "¿Hay errores de TypeScript?"

├─ Observation: 5 errores encontrados
├─ Thought: "¿Cuáles son los errores?"

├─ Observation: TypeScript limpio ✓

└─ Observation: Build exitoso ✓

2️⃣ CHAIN-OF-THOUGHT (CoT): Descomposición Lógica (20%)
═══════════════════════════════════════════════════════════════════════════════

DEFINICIÓN
CoT descompone un problema complejo en pasos intermedios, explicitando
el razonamiento en cada etapa.

APLICACIÓN EN SOBREBUDGET

CASO 1: Lógica del Semáforo
Problema: ¿Cómo sé si una categoría está en rojo, amarillo o verde?

Descomposición (CoT):
├─ Paso 1: Definir el problema
│  ├─ Entrada: presupuesto mensual, cantidad gastada
│  └─ Salida: estado visual (verde/amarillo/rojo)
│
├─ Paso 2: Métrica
│  └─ Porcentaje = (gastado / presupuesto) * 100
│
├─ Paso 3: Umbrales
│  ├─ Verde: ≤ 70%
│  ├─ Amarillo: 70-100%
│  └─ Rojo: > 100%
│
├─ Paso 4: Fórmula
│  ├─ if (percentage <= 70%) return 'green'
│  ├─ else if (percentage <= 100%) return 'yellow'
│  └─ else return 'red'
│
└─ Paso 5: Validación
   ├─ $500 de $800 = 62.5% → verde ✓
   ├─ $800 de $800 = 100% → amarillo ✓
   └─ $900 de $800 = 112.5% → rojo ✓

CASO 2: Almacenamiento Versionado
Problema: ¿Cómo manejar cambios en la estructura de datos?

Descomposición (CoT):
├─ Paso 1: Identificar riesgo
│  └─ Si cambio estructura, datos viejos se rompen
│
├─ Paso 2: Solución: Versionado
│  └─ Patrón: "sobrebudget:VERSION:TYPE"
│
├─ Paso 3: Implementación
│  ├─ const VERSION = 'v1'
│  ├─ const CATEGORIES_KEY = `sobrebudget:${VERSION}:categories`
│  └─ const TRANSACTIONS_KEY = `sobrebudget:${VERSION}:transactions`
│
├─ Paso 4: Beneficios
│  ├─ Si necesito v2: cambio VERSION
│  ├─ Compatibilidad hacia atrás garantizada
│  └─ Migraciones automáticas posibles
│
└─ Paso 5: Resultado
   └─ Sistema robusto y escalable ✓

CASO 3: Validaciones de Formularios
Problema: ¿Qué validaciones necesitan las transacciones?

Descomposición (CoT):
├─ Paso 1: Identificar campos críticos
│  ├─ Monto: sin número es inútil
│  ├─ Fecha: sin contexto temporal
│  ├─ Categoría: ambigüedad
│  └─ Tipo: confusión (ingreso vs gasto)
│
├─ Paso 2: Reglas
│  ├─ amount > 0 (no $0 ni negativos)
│  ├─ date requerido (no puede estar vacío)
│  ├─ categoryId requerido (no puede estar vacío)
│  ├─ type requerido (debe ser income o expense)
│  └─ note es opcional
│
├─ Paso 3: Implementación
│  ├─ if (!amount || amount <= 0) error
│  ├─ if (!date) error
│  ├─ if (!categoryId) error
│  └─ if (!type) error
│
├─ Paso 4: Testing
│  ├─ Intento guardar con amount=0 → Error ✓
│  ├─ Intento guardar sin categoría → Error ✓
│  ├─ Intento guardar sin nota → OK ✓
│  └─ Intento guardar válido → Guardado ✓
│
└─ Paso 5: Resultado
   └─ Datos siempre consistentes ✓

CASO 4: Filtrado Multi-Dimensional
Problema: ¿Cómo filtro transacciones por mes + categoría + búsqueda?

Descomposición (CoT):
├─ Paso 1: Dimensiones
│  ├─ Mes (YYYY-MM)
│  ├─ Categoría (específica o todas)
│  └─ Nota (búsqueda por texto)
│
├─ Paso 2: Lógica
│  └─ AND entre todos (debe cumplir TODOS los filtros)
│
├─ Paso 3: Algoritmo
│  ├─ for each transaction:
│  │  ├─ if (mes match) AND (categoría match) AND (texto match)
│  │  └─ include in result
│  └─ return filtered array
│
├─ Paso 4: Casos de Prueba
│  ├─ Mes="2024-01", Cat="todas", Texto="" → Enero completo ✓
│  ├─ Mes="2024-01", Cat="alimentación", Texto="" → Solo alimentación enero ✓
│  ├─ Mes="", Cat="", Texto="super" → Todas con "super" ✓
│  └─ Mes="", Cat="", Texto="" → Todas ✓
│
└─ Paso 5: Optimización
   └─ O(n) donde n = # transacciones ✓

DESCOMPOSICIONES CoT: 4 casos

✅ Explicó lógica compleja claramente
✅ Facilitó implementación correcta
✅ Documentó razonamiento para mantenimiento
✅ Permitió validación de correctitud


3️⃣ TREE-OF-THOUGHTS (ToT): Exploración Estratégica (10%)
═══════════════════════════════════════════════════════════════════════════════

DEFINICIÓN
ToT explora múltiples rutas de pensamiento, evaluando cuál es la mejor
antes de ejecutar. Es exploración en árbol de decisiones.

APLICACIÓN EN SOBREBUDGET

DECISIÓN 1: Almacenamiento (localStorage vs Supabase vs IndexedDB)
Pregunta: ¿Cómo persisto datos sin backend?

Árbol de Decisión:
├─ Rama A: localStorage
│  ├─ Pros:  ✅ Sin backend, sin .env, privacidad, offline
│  ├─ Cons:  ❌ No multiplataforma, ~5-10MB límite
│  ├─ Alineación: CUMPLE requisito "sin backend"
│  ├─ Score:  9/10
│  └─ Decisión: ⭐ SELECCIONADA
│
├─ Rama B: Supabase
│  ├─ Pros:  ✅ Profesional, multiplataforma, sync
│  ├─ Cons:  ❌ Requiere .env, requiere backend
│  ├─ Alineación: NO cumple requisito "sin backend"
│  ├─ Score:  3/10
│  └─ Decisión: ❌ RECHAZADA
│
└─ Rama C: IndexedDB
   ├─ Pros:  ✅ Más capacidad (>50MB)
   ├─ Cons:  ❌ Complejo, innecesario, browser-specific
   ├─ Alineación: Cumple pero overkill
   ├─ Score:  4/10
   └─ Decisión: ⚠️ NO NECESARIA

DECISIÓN 2: Gráficas (Recharts vs Chart.js vs D3.js vs Visx)
Pregunta: ¿Cómo muestro gráficas interactivas?

Árbol de Decisión:
├─ Rama A: Recharts
│  ├─ Pros:  ✅ React nativo, TypeScript, pequeño bundle
│  ├─ Cons:  ❌ Menos customizable que D3
│  ├─ Score:  9/10
│  └─ Decisión: ⭐ SELECCIONADA
│
├─ Rama B: Chart.js
│  ├─ Pros:  ✅ Popular, customizable
│  ├─ Cons:  ❌ No es React, API no-idiomática
│  ├─ Score:  6/10
│  └─ Decisión: ⚠️ Funciona pero no ideal
│
├─ Rama C: D3.js
│  ├─ Pros:  ✅ Ultra customizable, profesional
│  ├─ Cons:  ❌ Curva aprendizaje alta, bundle grande, verbose
│  ├─ Score:  2/10
│  └─ Decisión: ❌ Overkill
│
└─ Rama D: Visx
   ├─ Pros:  ✅ D3 idiomático en React
   ├─ Cons:  ❌ Aún más complejo que Recharts
   ├─ Score:  7/10
   └─ Decisión: ⚠️ Buena pero innecesaria

DECISIÓN 3: Estructura de Tipos
Pregunta: ¿Dónde y cómo organizo los tipos TypeScript?

Árbol de Decisión:
├─ Rama A: Todo en src/types/index.ts
│  ├─ Pros:  ✅ Simple, fácil de encontrar, centralizado
│  ├─ Cons:  ❌ Archivo puede crecer
│  ├─ Score:  9/10
│  └─ Decisión: ⭐ SELECCIONADA (proyecto pequeño)
│
├─ Rama B: Múltiples archivos (types/categories.ts, etc)
│  ├─ Pros:  ✅ Organizado por dominio, escalable
│  ├─ Cons:  ❌ Más importes, overkill para pequeño proyecto
│  ├─ Score:  5/10
│  └─ Decisión: ⚠️ Mejor para grandes proyectos
│
└─ Rama C: Inline (tipos junto a componentes)
   ├─ Pros:  ✅ Colocation lógica
   ├─ Cons:  ❌ Duplicación, difícil reutilizar
   ├─ Score:  2/10
   └─ Decisión: ❌ No recomendado

DECISIÓN 4: Sistema de Export
Pregunta: ¿Cómo respaldan usuarios sus datos?

Árbol de Decisión:
├─ Rama A: JSON download
│  ├─ Pros:  ✅ Universal, portable, simple, verificable
│  ├─ Cons:  ❌ Manual, no automático
│  ├─ Score:  9/10
│  └─ Decisión: ⭐ SELECCIONADA
│
├─ Rama B: Cloud sync (Dropbox, Google Drive)
│  ├─ Pros:  ✅ Automático, multiplataforma
│  ├─ Cons:  ❌ Requiere API keys, backend, complejidad
│  ├─ Score:  2/10
│  └─ Decisión: ❌ Fuera de scope
│
└─ Rama C: URL encoding
   ├─ Pros:  ✅ Shareable
   ├─ Cons:  ❌ URLs extremadamente largas, límite de caracteres
   ├─ Score:  3/10
   └─ Decisión: ❌ No práctico

DECISIÓN 5: Organización de Componentes
Pregunta: ¿Cómo organizo 6 componentes reutilizables?

Árbol de Decisión:
├─ Rama A: Globales + Específicos
│  ├─ Pros:  ✅ DRY, separación clara, escalable
│  ├─ Cons:  ⚠️ Más carpetas
│  ├─ Score:  9/10
│  └─ Decisión: ⭐ SELECCIONADA
│
├─ Rama B: Anidados por página
│  ├─ Pros:  ✅ Colocation lógica
│  ├─ Cons:  ❌ Mucha duplicación
│  ├─ Score:  5/10
│  └─ Decisión: ⚠️ Si hay duplicación
│
└─ Rama C: Todos en components/
   ├─ Pros:  ✅ Simple al inicio
   ├─ Cons:  ❌ Caótico rápido, 50+ archivos
   ├─ Score:  3/10
   └─ Decisión: ❌ No escalable

DECISIONES CORRECTAS: 5/5 = 100%
